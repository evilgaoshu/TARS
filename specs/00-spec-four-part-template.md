# TARS — 四段式 Spec 模板

> **状态**: Active Template
> **适用范围**: 面向产品对象、控制面页面、工作台页面、主菜单模块的 spec 编写
> **目标结构**: `功能 Spec + UX Spec + 内容 Spec + 验收清单`

---

## 1. 为什么改成四段式

原来的 spec 往往把对象、字段、状态、页面、文案、验收混在一起。

结果就是：

- 功能规则写清楚了，但页面仍然不好用
- UX 方向有了，但文案和标题还在讲旧故事
- 页面做完了，却没有统一的验收方式

四段式模板的目标不是把 spec 写得更长，而是把下面四件事强制拆开：

1. 这个对象或页面到底是什么
2. 用户在页面里要怎么完成任务
3. 页面上的文案和命名应该怎么讲
4. 怎么证明已经做对

---

## 2. 适用与不适用

### 2.1 适用

默认适用于：

- `20-component-*` 组件对象规范
- `40-web-console-*` 工作台 / 页面规范
- 面向用户可见控制面的对象域说明

### 2.2 不适用

下列文档不强制四段式：

- 纯架构设计
- 协议策略
- 底层运行时机制
- 测试策略
- 部署 / 依赖 / 运维手册

这类文档可以继续按更适合自己的结构编写。

---

## 3. 标准结构

每份适用的 spec，默认按以下四段组织：

1. `功能 Spec`
2. `UX Spec`
3. `内容 Spec`
4. `验收清单`

可以在这四段内继续分小节，但不要跳过这四类信息。

---

## 4. 第一段：功能 Spec

这一段回答的是：

- 这个对象 / 页面是什么
- 它不是什么
- 它有哪些关键状态、字段、约束和规则

建议内容：

### 4.1 对象定义

- 对象是什么
- 对象不是什么
- 当前心智边界

### 4.2 用户目标或平台目标

- 这个对象服务什么业务目的
- 高频任务是什么

### 4.3 状态模型

- 生命周期状态
- 健康 / 配置 / 运行时状态
- 状态优先级

### 4.4 核心字段与层级

- 关键字段
- 哪些是主对象字段
- 哪些是衍生字段
- 哪些是系统隐藏字段

### 4.5 关键规则与约束

- 主约束
- 边界条件
- 兼容规则
- 不允许的行为

---

## 5. 第二段：UX Spec

这一段回答的是：

- 用户如何在页面里完成任务
- 页面入口和 IA 应该怎样组织
- 默认看到什么、下一步怎么走

建议内容：

### 5.1 用户任务

- 高频任务
- 次级任务
- 首屏必须先回答的 3 个问题

### 5.2 入口与页面归属

- 主菜单归属
- 页面路径
- 列表 / 详情 / 创建 / 编辑 / 修复页分工

### 5.3 页面结构

- 列表页需要回答什么
- 详情页首屏优先级
- 创建流是单页还是多步
- 编辑流、修复流如何组织

### 5.4 CTA 与操作层级

- 主 CTA
- 次 CTA
- 高风险动作如何下沉

### 5.5 字段分层

- `L1` 默认可见
- `L2` 条件显示
- `L3` 高级区
- `L4` 实现细节默认隐藏

---

## 6. 第三段：内容 Spec

这一段回答的是：

- 页面标题、按钮、说明、空态、错误态应该怎么讲
- 哪些词可以用，哪些词不能用

建议内容：

### 6.1 命名

- 模块名
- 页面名
- 对象名
- 必须统一的术语

### 6.2 页面标题与副标题

- 列表页标题
- 详情页标题
- 创建 / 编辑标题
- 页面说明语气

### 6.3 CTA 文案

- 主按钮文案
- 次按钮文案
- 测试、保存、启用、修复等动作的统一措辞

### 6.4 状态文案

- empty
- error
- degraded
- disabled
- success / tested / saved

### 6.5 术语黑名单

明确哪些词不应该直接出现在主路径：

- raw DTO 名
- manifest / registry / internal token
- 兼容层旧命名
- 只有工程师看得懂的实现词

---

## 7. 第四段：验收清单

这一段回答的是：

- 做到什么程度才算这个 spec 被正确实现

建议内容：

### 7.1 页面级验收

- 页面标题和对象边界正确
- 首屏先回答用户任务
- 主 CTA 清楚
- 默认区没有不该暴露的实现细节

### 7.2 交互级验收

- 创建流合理
- 编辑流合理
- CSV / raw token / 手输对象 ID 是否已消除
- empty / error / degraded 状态是否清楚

### 7.3 测试要求

- 页面级测试
- 定向交互测试
- 浏览器或截图验收

### 7.4 剩余限制

- 哪些点可由前端独立完成
- 哪些仍依赖后端支持
- 哪些留到下一阶段

---

## 8. 推荐写法

### 8.1 优先表格化、清单化

不要把四段都写成长篇散文。

推荐：

- 状态模型用表格
- 字段层级用表格
- CTA / 状态文案用清单
- 验收项用 checkbox 风格清单

### 8.2 避免四段重复

同一件事不要在四段里重复讲四遍。

例如：

- 字段定义写在功能 Spec
- 字段默认展示层级写在 UX Spec
- 字段显示名称写在内容 Spec
- 字段展示是否正确写在验收清单

### 8.3 工作台页也使用四段式

对于 Dashboard / Sessions / Executions / Inbox 这类工作台页：

- `功能 Spec` 描述的是“工作台域”
- 不是假装把它写成单一配置对象

---

## 9. 推荐骨架

```md
# 模块名

## 1. 功能 Spec
- 对象定义
- 边界 / 非目标
- 状态模型
- 核心字段
- 关键规则

## 2. UX Spec
- 用户任务
- 页面入口与 IA
- 列表 / 详情 / 创建 / 编辑 / 修复流
- CTA 层级
- L1/L2/L3/L4 字段分层

## 3. 内容 Spec
- 页面标题与副标题
- 模块命名与术语
- CTA 文案
- 空态 / 错误态 / degraded 文案
- 不应出现的旧词或实现词

## 4. 验收清单
- 页面级验收
- 交互级验收
- 测试要求
- 浏览器 / 截图验收
- 剩余限制说明
```

## 11. 迁移原则

这次不要求一次性重写全部 spec。

推荐迁移顺序：

1. 新增 spec 直接使用四段式
2. 高频主菜单模块优先迁移
3. 当某个模块开始大改时，再顺手把对应 spec 改成四段式
4. 不为了“模板统一”去机械改写暂时稳定、且短期不变的底层策略文档
