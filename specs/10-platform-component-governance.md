# TARS — 平台一级组件统一治理基线

> 关联总览：[10-platform-components.md](./10-platform-components.md)

## 1. 目标

定义所有一级组件共享的治理约束，避免每个对象页各自长出一套生命周期、导入导出、审计和控制面模型。

## 2. 一级组件之间的关系

- `Connectors` 负责接外部系统，`Skills` 负责编排能力，两者同级。
- `Providers`、`Channels`、`Automations`、`Agent Roles` 都是一级对象，不应长期退化成配置文件片段。
- `Identity / Users / Authentication / Authorization` 是治理底座，不是附属页面。
- `People` 负责人员画像与业务归属，不等同于登录账号。

## 3. 控制面统一要求

- 对象配置归对象页，治理配置归治理页，平台总控归 `Ops`。
- 详情页默认服务“看状态、改配置、做验证”，不要退化成 manifest 编辑器。
- 列表页优先支持发现问题、筛选状态和进入详情，而不是堆字段。
- 高风险和低频动作下沉到“高级动作”或“开发者信息”，不抢主操作位。

## 4. 审计统一要求

- 所有一级组件的 create / update / enable / disable / import / export / rollback 都应留下审计记录。
- 审计记录要绑定操作者、时间、对象 ID、动作类型和关键差异摘要。
- 高风险能力触发审批时，审计链必须能解释“谁申请、谁批准、谁执行、结果如何”。

## 5. 导入导出统一要求

- 日常对象配置不应依赖 `Ops` 原始导入导出。
- 导入导出是高级能力，放在对象页次级动作或 `Ops` 高级模式。
- Bundle / module export / import 的统一规则见 [30-strategy-platform-config-bundles.md](./30-strategy-platform-config-bundles.md)。

## 6. 版本治理统一要求

- 不是所有一级组件都默认暴露 revision history，但所有支持升级/回滚的对象都应有统一心智。
- 日常页只展示“当前版本 / 是否有升级风险 / 是否可回滚”。
- 详细版本历史和回滚路径下沉到高级动作。

## 7. 实施原则

- 新增一级组件时，先对齐对象心智、状态模型、主操作和验证入口，再补页面。
- 如果一个对象只有配置编辑器、没有运行状态和验证入口，它还不算成熟的平台对象。
- 如果一个对象只能在 `Ops` 中管理，它应视为“尚未完成产品化”。

## 8. 与现有实现的关系

- 当前已经比较成熟的对象：`Connectors`、`Skills`、`Identity / Access`、`Observability`。
- 当前最容易重新长乱的区域：对象配置入口、导入导出、平台动作、版本治理。
- 本文档用于约束后续一级组件继续扩张时的统一方向。
