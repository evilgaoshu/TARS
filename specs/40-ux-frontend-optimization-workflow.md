# TARS — 前端 UI/UX 优化工作流

> **状态**: Active
> **适用范围**: Web Console 主菜单模块、详情页、创建流、编辑流、空态与错误态
> **关联**: [40-ux-design-system.md](./40-ux-design-system.md)、[00-spec-four-part-template.md](./00-spec-four-part-template.md)、[00-frontend-module-ui-ux-convergence-template.md](./00-frontend-module-ui-ux-convergence-template.md)、[00-nav-page-to-spec-map.md](./00-nav-page-to-spec-map.md)

---

## 1. 目标

这份工作流解决的问题不是“spec 不够细”，而是：

- spec 已经足够细，但页面仍然不好懂
- 字段和状态都对了，用户还是不知道下一步做什么
- 页面看起来完成了，但内容、排版、节奏、层级仍像后台拼装页

因此，前端优化不能只等价为“继续补 spec”或“继续对字段”。  
它必须把对象边界、页面叙事、内容设计、排版层级、交互路径和验证方式一起收口。

---

## 2. 核心原则

### 2.1 先优化用户心智，再优化字段

页面首先要让用户明白：

- 这是哪个对象
- 现在是什么状态
- 最常见的任务是什么
- 下一步该做什么

如果这些不清楚，就算字段和接口都对，页面仍然是不合格的。

### 2.2 先看真实页面，再看代码

每次优化必须从真实页面开始，而不是从 DTO、types、API schema 开始。

顺序固定为：

1. 看真实页面
2. 对照 spec / PRD / technical design
3. 明确用户任务和页面契约
4. 再修改实现

### 2.3 默认区只讲 L1 / L2

前端最常见的问题不是“缺字段”，而是“把不该默认出现的字段摆上来了”。

必须遵守：

- `L1`: 默认可见，直接支撑主任务
- `L2`: 条件显示，支撑当前上下文
- `L3`: 高级区或次级区
- `L4`: 系统实现细节，不应默认暴露

### 2.4 一个页面只允许一个主叙事

页面不能同时讲多个对象，也不能同时讲“对象本身”和“底层实现”。

如果一个页面同时在讲：

- provider 和 model binding
- connector 对象和 raw manifest
- automation 对象和 trigger 内部机制

那就是对象边界未收口。

### 2.5 验收以页面效果为主，不以代码解释为主

代码说明“为什么现在这样”，但不能代替“用户现在看到什么”。

每次优化完成时，必须能用页面级证据证明：

- 页面更清楚了
- 主任务更顺手了
- 默认区更克制了
- 文案和 CTA 更统一了

## 3. 默认工作流程

### 3.1 选模块

一次只处理一个模块，或一个明确的模块组。

推荐批次：

1. 全局壳：导航、标题、面包屑、全局搜索、页面头部
2. 高频工作台：Dashboard、Sessions、Executions、Inbox、Chat
3. 高配置风险：Providers、Channels、Connectors、Agent Roles、Notification Templates
4. 平台扩展：Automations、Skills、Extensions、Knowledge
5. 治理与运维：Audit、Logs、Outbox、Ops、Governance Rules

### 3.2 先做页面验收

在开始改代码前，必须先回答：

- 用户从侧边栏点进来会看到什么
- 首屏在讲什么
- 主 CTA 是什么
- 首屏有没有先回答“现在如何 / 下一步做什么”
- 是否暴露了不该默认暴露的实现细节

### 3.3 写页面契约

每个模块在实施前都必须先整理出页面契约，至少包含：

- 对象定义
- 高频任务
- 首屏必须回答的 3 个问题
- 页面主 CTA / 次 CTA
- 列表 / 详情 / 创建 / 编辑 / 修复动作分布
- `L1/L2/L3/L4` 字段层级
- 空态 / 错误态 / degraded 态

页面契约与模块 spec 默认按 [00-spec-four-part-template.md](./00-spec-four-part-template.md) 组织；逐模块页面收口记录使用 [00-frontend-module-ui-ux-convergence-template.md](./00-frontend-module-ui-ux-convergence-template.md)。

### 3.4 按三类问题拆改动

每轮优化都先把问题分成三类：

- `Copy / Display`
  - 标题、说明、badge、helper text、空态文案、字段命名
- `Interaction`
  - 创建流、编辑流、结构化输入、主次操作、折叠与高级区
- `IA / Boundary`
  - 导航归属、页面入口、模块边界、对象叙事、主菜单分组

这样可以避免把所有问题一起混改。

### 3.5 先改低风险高收益项

默认顺序：

1. 命名、标题、说明文案
2. 字段分层和默认区内容
3. CTA 层级与交互节奏
4. 结构化输入替代 CSV / raw string
5. 导航与 IA 调整

### 3.6 每轮都补页面级验证

至少补一项：

- 页面级 React 测试
- 浏览器检查 / 截图验收
- 定向交互回归测试

只跑类型检查和构建，不算 UI/UX 优化完成。

## 4. 团队角色分工

### 4.1 Spec / Product Owner

负责：

- 确认对象边界
- 确认页面首屏主问题
- 决定主 CTA 与任务优先级
- 确认哪些字段应属于 L1/L2/L3/L4

### 4.2 UX Reviewer

负责：

- 判断页面叙事是否清楚
- 判断内容层级是否合理
- 判断默认区是否仍有实现细节外溢
- 判断单页 / 多步 / 折叠 / 分组是否合适

### 4.3 Frontend Implementer

负责：

- 复用现有组件体系落地
- 避免新增第二套设计语言
- 补测试与浏览器验证
- 对后端未支持能力使用真实降级展示，而不是伪装支持

---

## 5. 每个模块必须交付的产物

每个模块优化完成时，至少要有以下四项：

1. 模块页面契约
2. 改动后的实现
3. 页面级验证证据
4. 剩余限制说明

### 5.1 模块页面契约

使用模板文档，至少包含：

- 模块卡片
- 用户任务
- 边界检查
- IA 检查
- 字段分层
- 交互检查
- 状态检查

### 5.2 实现改动

只做为当前模块服务的最小必要改动：

- 文案
- 结构
- 组件分层
- 页面骨架
- 交互与测试

### 5.3 页面级验证证据

至少满足其一：

- 浏览器截图
- 浏览器自动化快照
- 定向页面测试

### 5.4 剩余限制说明

必须明确写出：

- 哪些问题前端已独立解决
- 哪些问题受后端约束无法彻底解决
- 哪些问题属于下一轮 IA 或平台级调整

## 6. 常见反模式

### 6.1 DTO 驱动页面

表现：

- 页面围绕接口字段展开
- 文案像 schema 描述
- 用户必须理解内部 token 才能操作

处理方式：

- 回到对象和任务
- 先重做字段层级，再决定 UI

### 6.2 组件复用绑架产品体验

表现：

- 创建页只是编辑器的一个变体
- 一套后台表单硬套多个对象
- 页面为了复用组件牺牲主任务路径

处理方式：

- 允许页面骨架分离
- 复用字段组件，不复用错误的页面心智

### 6.3 把高级字段默认上浮

表现：

- raw config、manifest、marketplace metadata、secret refs 出现在默认区
- 用户第一次打开就面对大量系统字段

处理方式：

- 严格做 L1/L2/L3/L4 裁剪
- 默认区只保留主任务相关信息

### 6.4 用“可以先这样”长期替代正确 IA

表现：

- 页面标题和对象边界不一致
- 导航分组是历史遗留
- 新功能不断堆进错误模块

处理方式：

- 每轮至少收掉一个 IA 真问题
- 不让错误边界无限累积

---

## 7. 验收清单

模块完成前，必须逐项确认：

- 页面标题是否是对象名，而不是内部实现名
- 首屏是否先回答用户任务
- 是否只有一个主 CTA
- 默认区是否只保留 L1/L2
- 是否移除了 CSV / raw token / 兼容层暴露
- empty / error / degraded / disabled 是否都清楚
- 测试和浏览器验证是否补齐
- 是否写清剩余限制

---

## 8. 推荐节奏

### 8.1 单模块节奏

推荐：

1. 半天完成页面审查和页面契约
2. 半天到一天完成 P1 / P2 改动
3. 半天完成测试、浏览器验收和收尾

### 8.2 多模块批量节奏

推荐按主题批量推进，而不是按页面零散修补：

- 全局壳一批
- 高频工作台一批
- 高风险配置页一批
- 治理 / 运维页一批

## 10. Definition Of Done

团队层面的 UI/UX 优化，只有在下面条件都成立时，才算真正完成：

- 主要模块都已有页面契约
- 主菜单模块不再依赖旧式 gap 文档推进
- 每轮优化都以页面级证据收尾
- 内容、排版、交互、IA 的问题被分层处理，而不是混在一起
- 前端团队已经形成统一的“先页面、后代码、再验证”的工作方式
