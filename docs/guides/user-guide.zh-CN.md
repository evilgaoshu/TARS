> **版本**: v1.0

## 1. 产品简介

### 1.1 什么是 TARS

TARS (Troubleshooting & Automated Response System) 是一个面向告警分析与受控执行的 AIOps 智能运维助手。

### 1.2 核心价值

| 价值 | 说明 |
|------|------|
| 降低 MTTR | 自动收集指标和知识上下文，缩短从告警到诊断建议的时间 |
| 保持可控 | 命令执行必须经过人工审批，审计完整可追溯 |
| 形成复用 | 将闭环记录沉淀为知识，为后续复用做准备 |

### 1.3 核心流程

```
告警进入 → 自动收集上下文 → AI 诊断建议 → 人工审批 → 受控执行 → 结果回传 → 沉淀经验
```

### 1.4 适用场景

- **高频告警处理**: CPU、内存、磁盘、服务不可用等常见告警
- **标准化故障处理**: 已有 Runbook 或 SOP 的故障场景
- **值班场景**: 快速获取诊断建议和执行上下文

## 2. 用户角色和权限

### 2.1 角色定义

| 角色 | 职责 | 权限 |
|------|------|------|
| 值班 SRE/运维工程师 | 接收告警、查看诊断建议、发起审批请求 | 查看会话、发起执行请求 |
| 审批人 | 对执行请求进行人工审批 | 批准/拒绝/修改执行请求 |
| 平台管理员 | 部署系统、配置模型和渠道、查看运行状态 | 系统配置、监控查看 |
| 知识管理员 | 管理文档和知识沉淀 | 知识库管理 |

### 2.2 审批权限

| 风险级别 | 定义 | 审批要求 |
|----------|------|----------|
| `info` | 只读命令 | 单人审批 |
| `warning` | 可恢复写操作 | 单人审批 + 命令确认 |
| `critical` | 高影响或不可逆操作 | 双人审批 |

## 3. 告警处理流程

### 3.1 流程概览

1. **告警接收**: VMAlert 发送告警到 TARS
2. **自动分析**: TARS 自动收集指标上下文并生成 AI 诊断建议
3. **人工确认**: 值班人员收到 Telegram 消息，查看建议
4. **审批执行**: 如需执行命令，发起审批请求
5. **执行结果**: 命令执行完成后，结果回传到会话

### 3.2 会话状态说明

| 状态 | 说明 |
|------|------|
| `open` | 会话刚创建，等待分析 |
| `analyzing` | 正在进行 AI 诊断分析 |
| `pending_approval` | 等待人工审批执行请求 |
| `executing` | 正在执行命令 |
| `verifying` | 执行完成，正在验证恢复状态 |
| `resolved` | 会话已解决 |
| `failed` | 执行失败或超时 |

## 4. Telegram 交互指南

### 4.1 首次使用

1. **添加 Bot**: 在 Telegram 中搜索并添加 TARS Bot
2. **获取 Chat ID**: 将 Chat ID 提供给管理员配置到告警路由
3. **确认接收**: 等待管理员发送测试消息确认配置正确

### 4.2 接收诊断消息

当告警触发时，你会在 Telegram 收到包含以下信息的消息：

- **告警标题**: 告警摘要信息
- **告警详情**: 指标值、阈值、持续时间等
- **AI 诊断建议**: 自动生成的分析和建议命令
- **相关上下文**: 关联的指标查询结果

### 4.3 审批操作

当需要执行命令时，TARS 会发送审批请求消息，包含：

- 目标主机信息
- 风险级别 (info/warning/critical)
- 建议执行的命令
- 执行原因说明
- 回滚提示
- 审批时限

**可选操作**:

| 操作 | 说明 |
|------|------|
| ✅ 批准执行 | 同意执行建议的命令 |
| ❌ 拒绝执行 | 拒绝本次执行，返回分析状态 |
| ✏️ 修改后批准 | 修改命令内容后批准执行 |
| 🔄 转交他人 | 将审批请求转交给其他审批人 |
| ❓ 要求补充信息 | 请求补充上下文信息 |

### 4.4 对话模式

在会话过程中，你可以：

1. **追问信息**: 发送消息询问更多上下文
2. **查看状态**: 请求查看当前会话状态
3. **人工接管**: 明确表示要人工处理，AI 停止自动建议

### 4.5 常用命令

```
/status      - 查看当前会话状态
/history     - 查看会话历史
/help        - 显示帮助信息
```

## 5. Web Console 使用指南

### 5.1 访问 Web Console

1. 打开浏览器访问 `https://tars.example.com` (由管理员提供的实际地址)
2. 如需认证，输入管理员分配的凭据

### 5.2 界面导航

Web Console 包含以下主要功能区域：

- **概览面板**: 显示系统状态和关键指标
- **会话列表**: 查看所有告警会话
- **执行记录**: 查看命令执行历史
- **知识库**: 查看沉淀的知识记录
- **审计日志**: 查看操作审计记录
- **配置管理**: 管理员进行系统配置

### 5.3 查看会话列表

1. 点击左侧导航栏 "Sessions"
2. 查看会话列表，包含：
   - 会话 ID
   - 告警来源和标题
   - 当前状态
   - 创建时间
   - 最后更新时间

3. 使用过滤器筛选：
   - 按状态筛选 (open/resolved/failed 等)
   - 按时间范围筛选
   - 按告警来源筛选

### 5.4 查看执行详情

点击会话进入详情页面：

- **告警信息**: 原始告警内容
- **诊断建议**: AI 生成的分析结果
- **执行请求**: 所有的执行请求记录
- **执行输出**: 命令执行的完整输出
- **时间线**: 会话事件时间线

### 5.5 Runtime Checks 测试

管理员可以使用 Web Console 进行系统测试：

1. **Setup Status**: 查看系统各组件健康状态
2. **Smoke Test**: 发送测试告警验证链路
   - 如仍未初始化，先完成 `/setup` 首次安装，再进入 "Runtime Checks" 页面
   - 点击 "Send Test Alert"
   - 检查是否收到 Telegram 消息
   - 验证诊断建议是否正常生成

## 6. 典型使用场景

### 6.1 场景一：CPU 告警处理

**场景描述**: 收到 CPU 使用率超过阈值的告警

**处理流程**:

1. 收到 Telegram 告警消息，包含：
   - 告警主机: `prod-web-01`
   - 当前 CPU: 95%
   - 持续时间: 5分钟

2. 查看 AI 诊断建议：
   - 建议查询 TOP 进程
   - 建议查看系统负载趋势
   - 建议检查是否有 OOM 事件

3. 确认建议合理，点击 "请求执行"

4. 收到审批请求，查看：
   - 命令: `top -bn1 | head -20`
   - 风险级别: info (只读)
   - 目标主机: `prod-web-01`

5. 批准执行，等待结果返回

6. 查看执行输出，确认高 CPU 进程

7. 根据输出决定下一步操作

### 6.2 场景二：服务不可用告警

**场景描述**: 收到服务不可用的告警

**处理流程**:

1. 收到告警，包含服务名称和实例信息

2. 查看 AI 建议的诊断步骤：
   - 检查服务状态
   - 查看服务日志
   - 检查端口监听状态

3. 按顺序执行只读诊断命令

4. 根据诊断结果，决定是否需要重启服务

5. 如需重启，发起 `systemctl restart` 执行请求

6. 等待审批人批准 (warning 级别需审批)

7. 执行完成后验证服务恢复

### 6.3 场景三：人工接管

**场景描述**: AI 无法给出有效建议，需要人工处理

**处理流程**:

1. 收到告警，但 AI 建议显示 "需要人工判断"

2. 点击 "人工接管" 或发送 `/manual` 命令

3. TARS 停止自动建议，等待人工操作

4. 通过 SSH 或其他方式登录目标主机

5. 手动排查问题

6. 在 Telegram 或 Web Console 记录处理结果

## 7. 常见问题解答 (FAQ)

### 7.1 告警接收问题

**Q: 为什么收不到告警消息？**

A: 请检查以下几点：
- 确认 Chat ID 已正确配置到 TARS
- 确认 Telegram Bot Token 配置正确
- 检查 VMAlert 是否将 webhook 发送到 TARS
- 查看 Web Console 的 Setup Status 确认 Telegram 组件健康

**Q: 收到的告警信息不完整？**

A: TARS 会对告警进行标准化处理，部分原始字段可能映射到 labels 或 annotations 中。查看 Web Console 的会话详情可以看到完整原始数据。

### 7.2 AI 诊断问题

**Q: 为什么 AI 没有给出诊断建议？**

A: 可能原因：
- 模型服务不可用或超时
- 脱敏后的信息不足以生成建议
- 该告警类型没有匹配的知识文档
- TARS 当前处于 diagnosis_only 模式

**Q: AI 建议的命令不合适怎么办？**

A: 可以：
- 拒绝当前建议，重新分析
- 选择 "修改后批准"，调整命令内容
- 使用人工接管模式

### 7.3 审批执行问题

**Q: 审批请求发给了谁？**

A: 根据审批路由配置，默认按服务 owner 或值班组路由。审批消息中会显示当前审批人来源。

**Q: 审批超时了怎么办？**

A: 默认审批超时时间为 15 分钟。超时后会通知升级人。如需延长，可配置新的审批路由。

**Q: 为什么我的审批请求需要双人审批？**

A: Critical 级别的命令默认需要双人审批。这是安全策略，请联系第二位审批人。

### 7.4 执行失败问题

**Q: 命令执行失败，如何排查？**

A: 在 Web Console 查看执行详情：
- 检查错误信息
- 确认 SSH 连接是否正常
- 验证命令语法是否正确
- 查看目标主机日志

**Q: 执行输出被截断了？**

A: TARS 对执行输出有大小限制（默认 256KB），超过部分会被截断。如需完整输出，请直接在目标主机上查看。

### 7.5 安全相关问题

**Q: 我的数据会被发送到外部模型吗？**

A: TARS 在发送给外部模型前会进行脱敏处理，包括：
- IP 地址、主机名、域名
- 密码、Token、Secret

敏感信息会被替换为占位符。

**Q: 哪些命令可以执行？**

A: 可执行命令受授权策略控制，包括：
- 白名单命令可直接执行
- 黑名单命令被拒绝
- 其他命令需要审批

具体配置请联系管理员。

### 7.6 其他问题

**Q: 如何查看历史告警记录？**

A: 登录 Web Console，进入 "Sessions" 页面，可按时间范围筛选历史记录。

**Q: 如何导出审计日志？**

A: 管理员可通过 Ops API 或 Web Console 导出审计日志。

**Q: 系统支持哪些告警源？**

A: MVP 版本支持 VMAlert Webhook。后续版本将支持更多告警源。
