# TARS 聚焦版定义：Incident Copilot

> 日期：2026-04-17  
> 状态：当前聚焦版产品定义  
> 目的：把 TARS 从“大而全的平台叙事”收窄成一个对值班排障真正有约束力的产品定义。

---

## 1. 一句话定义

TARS 是一个面向值班排障人的 incident copilot。  
它接住告警，优先用 `metrics -> logs -> traces` 自动取证，给出结论、风险和下一步建议；只有证据不足时，才后置进入 `SSH / 执行 / 审批`。

---

## 2. 核心用户

当前只服务一类核心用户：

- 值班排障的人

不是平台管理员优先，不是扩展开发者优先，也不是“所有运维对象都统一管理的人”优先。

---

## 3. 核心任务

当前阶段，TARS 最重要的任务不是“执行更多动作”，而是：

- 先确认问题是否真实存在
- 尽快定位最可能的根因
- 给出值班人下一步最可信的建议

一句话说，当前最该打透的是：

`确认问题 -> 找根因 -> 给出下一步`

---

## 4. 唯一黄金路径

当前产品只保留一条核心黄金路径：

1. 告警进入系统
2. 直接落到 `Session Detail`
3. 系统先自动取证
4. 第一屏先展示：
   - 当前结论
   - 风险等级
   - 下一步建议
   - 关键证据摘要
5. 若证据不足，系统继续补证据，并给人工排查方向
6. 只有在证据不足以确认根因时，才后置进入 `SSH / 受控执行`
7. 如需高危动作，必须进入审批
8. 修复后持续观察
9. 最终形成结案摘要与沉淀

---

## 5. 默认取证顺序

默认顺序固定为：

1. `metrics`
2. `logs`
3. `traces`
4. `SSH / execution`（仅当前三者不足时）

设计原则：

- 监控优先
- 只读优先
- 自动取证优先
- 执行后置
- 高危命令审批

这条顺序比“能上机就先上机”更符合值班排障的真实工作流，也更能体现 TARS 的产品价值。

---

## 6. Session Detail 是主舞台

`Session Detail` 是当前最重要的页面，不是辅助页面。

它应承担的角色是：

- 值班诊断首页
- 告警确认页
- 根因分析页
- 下一步建议页

它不应该首先表现为：

- 原始 JSON 浏览页
- 平台对象拼装页
- 审计日志堆叠页
- “你自己往下翻找事实”的页面

### 第一屏只保留两类内容

1. 结论区
   - 当前结论
   - 风险等级
   - 下一步建议
   - 当前状态（已确认 / 仍在补证据 / 建议人工介入）

2. 证据摘要区
   - Metrics 摘要
   - Logs 摘要
   - Traces 摘要（仅当有明确价值时显示）

以下内容必须下沉到第二屏以后：

- 完整 `tool_plan`
- 时间线细节
- 审计细节
- narrative 原文大段文本
- 原始上下文 JSON
- 附件原始列表

---

## 7. 页面优先级

当前默认心智应收敛为：

- `Sessions`：分诊队列
- `Session Detail`：主入口、主舞台
- `Executions`：仅当需要动作时进入

其他页面都不应抢主线：

- `Ops`
- `Connectors`
- `Automations`
- `Extensions`
- `MCP / Skill Source`
- `Setup`
- `Runtime Checks`

这些页面可以保留，但不应定义产品，不应成为值班用户默认心智入口。

---

## 8. 什么是当前不该继续扩的

以下方向当前都不应再抢主线：

- 为了“像平台”继续增加对象中心页面
- 让值班用户先理解 providers / connectors / roles / extensions / sources 的关系
- 把一次告警诊断拆散到多个页面完成
- 把重点从“结论与证据”转回“配置与治理”
- 为了抽象完整而继续扩大平台边界

这些能力不是不做，而是明确后置。

---

## 9. 当前明确后置的方向

以下方向保留，但属于后置层：

- 更完整的 connector 生命周期管理
- 更复杂的审批与治理流
- 平台级 dashboard / governance center
- MCP / Skill Source marketplace 叙事
- 更完整的扩展生态与插件管理
- 更丰富的后台平台对象管理

判断标准很简单：

如果一个能力不能直接强化

`告警 -> Session Detail -> 自动取证 -> 结论 -> 下一步`

那它就不应成为当前优先级中心。

---

## 10. 对当前主线的直接约束

接下来所有产品与前端判断，都应先回答这 5 个问题：

1. 它是不是直接服务值班排障的人？
2. 它是不是直接强化 `Session Detail` 主路径？
3. 它是不是直接提升“确认问题 / 找根因 / 给下一步建议”的质量？
4. 它是不是遵守 `metrics -> logs -> traces -> SSH` 的顺序？
5. 如果现在不做，会不会明显削弱 incident copilot 的核心体验？

只要连续两三个问题答“不”，就不应进入当前最高优先级。

---

## 11. 当前最该先做的三件事

1. 把 `Session Detail` 第一屏重构成真正的值班诊断首页
2. 把 `Tool-plan` 真正绑定到 `metrics -> logs -> traces -> SSH` 的取证顺序
3. 弱化平台对象页面在产品叙事中的存在感，让 `Sessions / Session Detail` 成为唯一主入口

---

## 12. 结论

TARS 当前不该继续追求“大而全”。

它应该先成为一个真正好用的 incident copilot：

- 告警来了，先接住
- 先查证据
- 先给判断
- 不轻易上机
- 不轻易执行
- 必要时再进入受控动作

先把这条主线做到小而美，其他平台化能力才值得继续长出来。
